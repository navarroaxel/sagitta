# Authoring example SVGs

These `*.svg` files are **hand-authored, static free-body diagrams** of structures
(trusses and frames): geometry + supports + applied loads + given support
reactions. They are illustrations, not output of the solver — nothing here is
generated by `src/lib`. Keep them self-contained (no CSS, no JS, no external refs)
so they render identically in a browser, in Markdown, and via `rsvg-convert`.

When asked to "create a new example based on examples/", copy the conventions
below. The cleanest models to clone from:

- **`l-frame-hinge.svg`** — L-frame: column + continuous beam, pinned + two
  rollers, internal hinge, a UDL, a concentrated moment, plus point loads
  (vertical + horizontal) at a node. Touches almost every building block.
- **`portal-frame.svg`** — solid-web portal, two pinned supports, internal hinge,
  inclined + vertical UDLs, a point load.
- **`simple-truss.svg`** — parallel-chord truss, roller + pinned, bar labels.
- **`frame-truss.svg`** — has the only **concentrated-moment** glyph.
- **`two-column-portal.svg`** — full two-column portal, **roller + pinned**; roof on
  two columns, horizontal point loads at both top corners and at mid-column, a vertical
  point load, a UDL, and a point load **hanging from the roof**. Units in **T** (not
  kN); both vertical reactions drawn below their supports and vertically aligned.
- **`l-frame-overhang.svg`** — L-frame with a **cantilever roof overhang**, pinned +
  roller, a UDL, a point load, a clockwise moment at the roller, and a horizontal load
  at the free tip drawn *outside* the node (collinearity exception). No hinge —
  determinate through the supports alone.
- **`fixed-beam-hinge-overhang.svg`** — straight horizontal beam with a **fixed
  (built-in) support**, an internal hinge, a roller, and an overhang; point loads, a
  UDL, and the **reaction moment** (M_A arc) at the fixed end. The canonical example
  for the fixed-support + reaction-moment building blocks (see also
  `t-frame-fixed.svg`, a column fixed at its base).

## Coordinate system

World is **metres, y-up**. SVG is **y-down**. Every file picks an affine transform
and documents it in the header comment:

```
Transform: sx = OX + x*k ; sy = OY - y*k
```

Pick `k` (px per metre, typ. **40–60**) and origin `(OX, OY)` so the structure
fills the canvas with margin for labels/arrows. Compute every screen coordinate
from this transform — do **not** eyeball points. Then set `viewBox` to the actual
content bounds (it need not start at `0 0`; see `portal-frame` and
`l-frame-hinge`) and draw a matching white background `<rect>` first.

## Header comment (required)

Open every file with an **English** comment block documenting the model, in this
order: **Structure**, **Transform**, **Nodes** (id, world coord → screen coord,
support type), **Members**, **Supports**, **Loads**, **Reactions**. State whether the
given reactions satisfy equilibrium; if they do not, say so and draw them as given
anyway (see the NOTE in `l-frame-hinge.svg`). Loads may be in **kN or T** — be
consistent within a file and label accordingly.

## Palette (do not invent colors)

| Role                         | Color     | Notes                                   |
| ---------------------------- | --------- | --------------------------------------- |
| Members / supports / nodes   | `#1a1a1a` | members `stroke-width:5` (frame) or `3` (truss), `stroke-linecap:round` |
| Fills (supports, hinge, bg)  | `#ffffff` |                                         |
| Node & bar labels            | `#6a1b9a` | bar labels also `font-style:italic`     |
| Dimension lines              | `#9aa0a6` | `stroke-width:1`                        |
| Dimension text & caption     | `#5f6368` |                                         |
| **Loads**                    | `#c0392b` | red — applied forces, moments, UDLs     |
| **Reactions**                | `#1a73e8` | blue — support reactions                |
| Tension (N>0) accent         | `#0b8a8a` | teal; rarely needed in these inputs     |

Font is always `system-ui, sans-serif` (set once on `<svg>`). Labels/loads/reactions:
`font-size:13`, `font-weight:600`–`700`. Dimensions: `font-size:12`. Subscripts use
`<tspan font-size="10" dy="4">A</tspan>` (and reset baseline with a following
`dy="-4"` tspan if more text follows). End the file with one English `<text>`
caption (`fill:#5f6368; font-size:13`) summarizing the figure.

## Layer order

Draw back-to-front, each concern in its own `<g>`: **dimensions → members →
supports → hinge → nodes → node/bar labels → loads (red) → reactions (blue) →
caption.** Don't merge concerns into one group.

## Building blocks

Coordinates below assume a node at `(NX, NY)`; translate the literals from a
canonical file. **Supports hang from the node** (apex AT the node, symbol below).

**Pinned support** — triangle + ground hatch:
```svg
<g stroke="#1a1a1a" stroke-width="2" fill="#ffffff">
  <polygon points="NX,NY  NX-9,NY+11  NX+9,NY+11"/>
  <line x1="NX-15" y1="NY+11" x2="NX+15" y2="NY+11"/>
  <!-- 4 diagonal hatch ticks from y=NY+11 down to y=NY+18 -->
</g>
```

**Roller support** — pinned triangle + two wheels + lower hatch
(`l-frame-hinge.svg`, support at B/C). Adds `<circle r="3">` wheels at
`NY+17` and the hatch at `NY+21`.

**Fixed / built-in support (empotramiento)** — a solid line *perpendicular to the
member* through the node, with short hatch ticks trailing off its far side (the
member runs away from it). Carries H, V **and moment**, so a fixed end usually also
gets a **reaction-moment arc** (see *Concentrated moment*). A beam fixed at its end
uses a **vertical** wall (`fixed-beam-hinge-overhang.svg`); a column fixed at its
base uses a **horizontal** ground line (`t-frame-fixed.svg`):
```svg
<!-- beam end fixed to a wall on the LEFT (member runs right); 5 hatch ticks, 10 px apart -->
<g stroke="#1a1a1a" stroke-width="2" fill="none">
  <line x1="NX" y1="NY-20" x2="NX" y2="NY+20"/>       <!-- wall through the node -->
  <line x1="NX" y1="NY-20" x2="NX-8" y2="NY-12"/>     <!-- ...repeat down to NY+20 -->
</g>
```

**Internal hinge** — white open circle on the member:
```svg
<circle cx="NX" cy="NY" r="6" fill="#ffffff" stroke="#1a1a1a" stroke-width="2"/>
```

**Rigid joint / node** — filled dot `<circle r="4" fill="#1a1a1a"/>`. (A hinge is
an *open* circle; a rigid corner is just the filled dot — that's the only visual
difference, so don't put a filled dot on top of a hinge.)

**Point load** — shaft + filled triangular arrowhead (`stroke:none`), tip at the
point of application, label offset clear of the shaft. See loads in `simple-truss.svg`.
For a load **hanging from a member** (a suspended weight), draw it on the far side of
the member: shaft starting at the member, arrowhead at the loose end pointing away
(see the `12 T` load in `two-column-portal.svg`). When an arrow stands in open space
(not piercing a support), let the shaft **overlap the arrowhead base by a few px** —
the arrowhead is drawn last and hides the join; a 2–3 px gap otherwise reads as a
split arrow (tip detached from the shaft).

**UDL** — a top rail line, evenly spaced vertical shafts, an arrowhead per shaft,
and a `q = … kN/m` (or `T/m`) label. Inclined UDLs (perpendicular to a member) exist
in `portal-frame.svg`; a partial-span horizontal UDL is in `two-column-portal.svg`.

**Concentrated moment** — arc + arrowhead + center dot (`frame-truss.svg`,
`l-frame-hinge.svg`). Clockwise uses an SVG arc with
`large-arc-flag=1 sweep-flag=1` and a small gap at the top where the arrowhead
sits; counterclockwise uses `sweep-flag=0`. To resize, scale the arc radius and the
arrowhead offsets together about the center. The **reaction moment** at a fixed
support (`M_A` in `fixed-beam-hinge-overhang.svg`) uses the same glyph drawn in the
reactions (blue) group.

## Reaction / load arrow convention

Reactions and loads are drawn **tip AT the node, shaft trailing in the opposite
direction** of the force (the arrowhead pushes on the node). A leftward force →
arrowhead on the left, shaft to the right; an upward reaction → arrowhead at the
node, shaft below (it pierces the support symbol — that's expected, blue over
black). Comments in the example files say `tip at the node, shaft trailing opposite`.

**Exception:** when trailing the shaft would make it collinear with a member (e.g.
a *downward* reaction at the base of a column, where the column is directly above),
draw the arrow just **outside** the support pointing in the force direction instead
— otherwise the arrow disappears into the member. See `V_A` in
`l-frame-hinge.svg`. Color (blue) + the `V_A`/`H_A` label keep it unambiguous.

## Before finishing — verify visually

Render and look at it; tight clusters (a node with a support + reaction + moment)
overlap easily.

```bash
rsvg-convert -w 1100 examples/your-file.svg -o /tmp/out.png   # then open the PNG
```

Check: nothing clipped by the `viewBox`; arrows point the right way; labels don't
collide; subscripts sit on the baseline; the white `<rect>` covers the whole frame.
